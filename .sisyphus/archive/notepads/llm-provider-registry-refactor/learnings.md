- Added a static LLM registry in `crypto_news_analyzer/config/llm_registry.py` as the sole validation source for provider names, official model names, provider auth metadata, and per-model capability flags.
- The new structured `llm_config` schema now requires object-based `model`, `fallback_models`, and `market_model` entries; legacy `summary_model` and the alias `kimi-for-coding` are rejected with migration-focused validation errors.
- `thinking_level` validation is centralized in the registry, limited to `disabled|low|medium|high|xhigh`, and fails fast when configured on unsupported models or when unknown model options are supplied.
- `tests/test_llm_analyzer.py` now initializes `LLMAnalyzer` with `provider_credentials` and registry-backed `llm_config` payloads (`model`, `fallback_models`, `market_model`) instead of removed `api_key`/`GROK_API_KEY` constructor parameters or legacy model aliases.
- Property-based config tests now derive valid model objects from `llm_registry.MODELS`, keep `fallback_models` as structured lists, and constrain generated REST API `response_mapping` field names to non-whitespace values so Hypothesis only emits validator-acceptable configs.
