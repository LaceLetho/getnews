# Create OpenClaw HTTP API Skill

## TL;DR
> **Summary**: Add a repo-local OpenClaw skill in idiomatic English that documents every supported live HTTP endpoint, explains each endpoint’s business meaning and usage workflow, and includes a maintainer playbook for keeping the skill synchronized with code and docs as the API evolves.
> **Deliverables**:
> - `.opencode/skills/crypto-news-http-api/SKILL.md`
> - `.opencode/skills/crypto-news-http-api/references/analyze-workflow.md`
> - `.opencode/skills/crypto-news-http-api/references/datasource-management.md`
> - `.opencode/skills/crypto-news-http-api/references/operations-and-maintenance.md`
> - Targeted tests and drift guards for structure, endpoint inventory, exclusions, and update guidance
> **Effort**: Medium
> **Parallel**: YES - 3 waves
> **Critical Path**: Task 1 → Task 2 → Task 5 → Tasks 6-9 → Task 10

## Context
### Original Request
- Build a skill for OpenClaw that explains the current project HTTP API usage.
- The skill must clearly describe each endpoint’s business meaning and usage.
- The skill must keep pace with future project evolution and API changes.
- The skill must explain how to upgrade/maintain itself.
- All skill content must be natural English.

### Interview Summary
- Scope is **all supported HTTP endpoints**, not just the analyze flow.
- Audience priority is **API users first**, with a dedicated maintainer/update path.
- Sync model is a **canonical-source workflow** tied to repo code/docs, not ad-hoc manual edits.
- Verification strategy is **tests-after plus stronger sync checks**.
- Default implementation shape is a repo-local skill under `.opencode/skills/crypto-news-http-api/` using the existing `SKILL.md` + `references/` pattern.

### Metis Review (gaps addressed)
- Added an explicit **source-precedence policy** so the executor knows what to trust when code, tests, and prose drift.
- Added a **Must NOT Document** boundary to prevent endpoint sprawl and Telegram-command scope creep.
- Required a **workflow-first structure** instead of an endpoint dump.
- Required explicit coverage of the async **job-state machine**, datasource secret redaction, and maintainer sync workflow.
- Tightened verification to fail on missing live endpoints, missing required sections, and unsupported/deprecated endpoint claims.

## Work Objectives
### Core Objective
Create a maintainable OpenClaw skill that lets an executor or AI agent correctly use this repository’s live HTTP API without guessing, while also giving maintainers an explicit procedure for updating the skill whenever routes, schemas, auth behavior, or operational notes change.

### Deliverables
- New skill directory: `.opencode/skills/crypto-news-http-api/`
- Main skill entrypoint with YAML front matter and high-level workflow guidance
- Detailed reference docs for analyze workflow, datasource management, and operational/maintenance topics
- New or updated automated tests validating structure, endpoint inventory, exclusions, required semantics, and update workflow
- Targeted verification run output showing the skill and sync checks pass

### Definition of Done (verifiable conditions with commands)
- `uv run pytest tests/test_opencode_skill_crypto_news_http_api.py -v`
- `uv run pytest tests/test_api_server.py -k "health or analyze or datasource or webhook" -v`
- `uv run pytest tests/test_banned_legacy_reference_scan.py -v`
- `uv run python tests/helpers/banned_legacy_reference_scan.py`
- Skill files exist only in English prose and document the supported endpoint inventory plus maintainer update workflow.

### Must Have
- YAML front matter matching repo skill conventions
- A quick-reference section for auth and endpoint inventory
- Workflow-first analyze guidance: create job → poll status → fetch result
- Clear endpoint business meaning, auth requirements, request/response expectations, and failure semantics
- Datasource CRUD coverage, including tag limits and redacted secret behavior
- Telegram webhook coverage labeled as maintainer/integration-oriented, not the primary API user path
- An explicit “Updating” workflow tied to canonical repo files/tests
- Tests that fail when the skill omits a supported endpoint or documents an excluded one as supported

### Must NOT Have (guardrails, AI slop patterns, scope boundaries)
- No Chinese text in the skill body or references
- No unsupported/deprecated compatibility surfaces presented as active APIs
- No autogenerated FastAPI/OpenAPI docs routes documented unless they are intentionally supported in repo code
- No Telegram slash-command tutorial inside this HTTP API skill
- No secret values, real bearer tokens, or unredacted datasource auth examples
- No prose-only “maintenance guidance” without concrete source files and verification commands
- No endpoint catalog that omits business meaning or workflow context

## Verification Strategy
> ZERO HUMAN INTERVENTION - all verification is agent-executed.
- Test decision: **tests-after** using `pytest`
- QA policy: Every task includes concrete agent-executed scenarios
- Evidence: `.sisyphus/evidence/task-{N}-{slug}.{ext}`
- Source precedence for sync enforcement:
  1. `crypto_news_analyzer/api_server.py` + `tests/test_api_server.py` = canonical live-route/auth/status-code truth
  2. `crypto_news_analyzer/datasource_payloads.py` + `crypto_news_analyzer/domain/models.py` = canonical datasource contract truth
  3. `docs/AI_ANALYZE_API_GUIDE.md` = canonical user-facing analyze narrative **unless** it conflicts with code/tests
- Known drift rule: if `AI_ANALYZE_API_GUIDE.md` conflicts with code/tests, the skill follows code/tests and the maintenance guidance must tell maintainers to reconcile the guide separately.
